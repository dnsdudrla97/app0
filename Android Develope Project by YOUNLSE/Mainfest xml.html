<?xml version="1.0" encoding="utf-8"?>
<manifest xmlns:android="http://schemas.android.com/apk/res/android"
package="com.sofsqared.yun">
<application
android:allowBackup="true" //앱 백업 서비스 지원 여부
android:icon="@mipmap/ic_launcher" //앱 설치시 나타낼 아이콘
android:label="@string/app_name" //프로젝트를 생성시 입력한 앱의 이름을 제시
android:roundIcon="@mipmap/ic_launcher_round" //아이콘이 원형으로 출력되는 상황을 제시
android:supportsRtl="true" //Right , Left 레이아웃 지원여부를 나타댄다.
android:theme="@style/AppTheme"> //앱의 테마를 설정 -> android:theme="@style/App Theme"
//테마는 앱에서 사용되는 모양, 색깔 등을 지정하므로 변경되면 전체적인 느낌이 변경
<activity android:name=".MainActivity">
<intent-filter>
<action android:name="android.intent.action.MAIN" /> 
//시작 액티비티를 지정하기 위한 액션을 지정한다.
<category android:name="android.intent.category.LAUNCHER" />
//인텐트 필터에 카테고리 이름을 추가한다. 
//애플리케이션에서 가장 먼저 실행되는 액티비티
</intent-filter>
</activity>
</application>
</manifest>

<activity android:name=".MainActivity">
  • activity는 화면에 가장 먼저 띄어주는 최상위 화면이며, 여기 메소드로 보면 Mainactivity를 가장 먼저 띄어준다.
  • App을 구성하고 있는 Component에 무엇무엇이 있는지, 또 어떤 기능을 하는지가 적혀있고 App이 잗동할때 필요한 권한들이 적혀있다고 한다.
  • Mainfest에 App, App's Component에 대한 구조와 meta-data에 정의가 되어있기 때문에 중요하다.

Mainfest

  1. application의 root directory에 있는 xml 파일
  1. App의 필수적인 정보를 안드로이드 시스템에 전달한다.
  1. 필수적인 정보는 App의 코드를 실행하는데 필요한 정보를 말한다.
  1. App을 위한 다른것들과 중복 X, java 패키지명을 정의한다.
  1. 앱을 구성하는 Component의 클래스명을 정의한다.
  1. 해당 Component가 처리하는 기능이 무엇인지 <intent-filter>를 통하여 정의한다.
  1. App에서 어떤 Component가 존재하고 어떤 조건에서 작동하는지 핸드폰이 알 수 있다.
  1. App이 핸드폰의 제한된 API에 접근하거나 다른 App의 Component를 사용하기 위한 권한을 설정한다.
  1. 모니터링 Instrumentation class를 설정 → 개발 종료시, 배포시 삭제

App Component App을 구성하는 Component 앱의 4대 구성요소

  • Activity
  • Service
  • <receiver> Broadcast Receiver
  • <provider> Content Provider
    • 이 컴포넌트 클래스가 앱에서 실행될려면 androidmainfest.xml에서 각각의 클래스에 해당하는 <activity>, <service>, <receiver>, <provider>가 선언되어야 한다.
Activity : UI를 지시하고 화면에 대한 상호작용을 처리한다.
Service : 백그라운드 프로세싱 관련 애플리케이션 처리
  • <receiver> BroadCast Receivers : Android os와 앱 사이의 통신을 처리한다.
  • <provider> Content Providers : 데이터및 데이터베이스 관리 문제를 처리한다.

MAINFEST

  • permission
    • 이 응용프로그램이나 다른 응용프로그램의 특정 구성 요소 나 기능에 대한 액세스 제한
  • uses-configuration
    • 응용프로그램이나 필요한 하드웨어 및 소프트웨어 기능을 나타낸다.
    • 실제 키보드나 특정 탐색 장치가 필요하다.
  • uses-sdk
    • 안드로이드 시스템의 API레벨과 비교를 통해 안드로이드 기기에서 서로 다를수 있다.
  • instrumentation
    • 계측기, 응용프로그램과 시스템의 상호작용을 모니터링 할 수 있는 클래스를 선언한다. 오브젝트는 응용프로그램의 구성 요소보다 먼더 인스턴스화 한다.
  • uses-feature
    • 단일 하드웨어, 소프트웨어 기능을 선언한다. 선언의 목적은 외부 엔터티에 애플리케이션이 의존하는 하드웨어 및 소프트웨어 기능 집합을 알려준다.
    • 이 요소는 required 특성을 제공한다. 이 특성을 통해 애플리케이션에 선언된 기능이 필요하고 해당 기능 없이는 작동할 수 없음을 지정
    • 해당 기능을 가지는 것을 선호하지만 해당 기능 없이는 작동할 수 있음을 지정할 수 있다.
    • 기기별 가변 기능을 설명하는 중요한 역활을 한다.
  • uses-permission
    • 앱 작동시 사용자가 반드시 부여해야 하는 시스템 권한이다.
  • uses-permission-sdk-23
    • 앱이 특정 권한을 원한다는 것을 지정한다.(안드로이드 6)이상 실행하는 기기에서 설치되는 경우에만 해당
    • 새 기능을 포함하기 위해 앱을 업데이트 하는 경우 유용한다.
  • permission-grou
    • 관련 권한의 논리적 그룹 이름을 선언한다.
    • 개별 권한은 <permission>요소의 속성을 통해 그룹에 가입한다.
    • 그룹의 구성원은 사용자 인터페이스에 함께 표시한다.
  • support-screens
    • 응용프로그램이 지원하는 화면 크기를 지정하고 응용프로그램이 지원하는 것보다 큰 화면에 대해 화면 호환성 모드를 활성화 할 수 있다.
    • 응용프로그램에서 항상이 요소를 사용하여 응용프로그램이 지원하는 화면 크기를 지정해야 한다.
  • compatible-screens
    • 호환 화면, 각 화면 구성을 지정한다.
  • support-gi-texture
    • 앱이 지원하는 단일 GL 텍스처 압축 형식을 선언한다.
    • 응용프로그램이 장치에 설치되면 해당 압축 된 텍스처 자신을 제공 할 수 있는 경우 응용 프로그램은 GL 텍스처 압축 형식 "지원" 한다.
    • 응용프로그램은 내부에서 압축된 자산을 로컬로 제공 하거나 런타임에서 다운로드 할 수 있다.
  • permission-tree
    • 권한 트리의 기본 이름을 선언한다.
    • 응용프로그램은 트리내의 모든 이름을 소유호출하여 트리에 새로운 권한을 동적으로 추가할 수 있다.

<mainfest><application> 이 무조건 포함되어 있어야한다.

<intent-filter>는 <action>, <data>, <category>를 포함해야한다.

<provider>는 <grant-uri-permission>, <path-permission>을 포함해야한다.

우선순위

  1. 첫번째 속성 : <mainfest>, <application>
  1. 두번째 속성 : <activitiy>, <activity-alias>, <provider>, <receiver>, <service>
  1. 세번째 속성 : <intent-filter>, <action>, <meta-data>
  1. 네번째 속성 : <permission>, <uses>
  1. 다섯번째 속성 : <tag>..

First priority

  • <mainfest> 는 xml 파일의 근본 요소이고 xmlns:android와 package의 속성을 지정한다.
  • xmlns:android
    • 안드로이드에 namespace를 정의하고 항상 http://schemas.android.com/apk/res/android"으로 설정해야한다고 한다.
  • package
    • 안드로이드에서 프로젝트를 생성할때 내가 생성한 package 이름이 나와있다. 중복불가
  • android:sharedUserId
    • 다른 앱과 공유할 리눅스 아이디의 이름이다.

서로다른 어플에서 동일하게 설정하면 데이터를 엑세스할 수 있고 원하는 경우 동일한 프로세스에서 실행될 수도 있다. → 인증서가 동일해야 한다

  • android:sharedUserLabel
    • 동일한 UserId를 사용할 때 앱을 구분하기 위한 닉네임
  • android:versionCode
    • 내부 버전 번호로 최신버전인지 아닌지를 확인하는데 사용한다.
  • android:versionName
    • 사용자에게 버전을 보여주는 용도
  • android:installLocation
    • 앱을 설치할때 내부 저장장치에만 할 수 있는지 외부 저장장치에도 할 수 있는지 설정한다.명령어 → internalOnly, auto, perferExternal

##<application>

  • 어플리케이션 선언문
    • 이 태그는 어플리케이션의 각 컴포넌트를 선언하는 하위요소를 포함(<activity>, <service>)
    • 모든 컴포넌트에 영향을 줄 수 있는 attribute를 가지고 있다.
    • 어플리케이션의 속성은 컴포넌트에 영향을 주는 것이고 이 컴포넌트들은 대부분 메인 컴포넌트를 의미한다
  • android:allowBackup
    • 앱이 백업 서비스 지원 b여부를 나타낸다.
  • android:icon
    • 앱을 섪치하면 나타낼 아이콘ㅇㅇ
  • android:label
    • 프로젝트를 생성할때 입력한 앱의 이름을 나타낸다. 앱 설치시 나타나는 이름
  • android:roundicon
    • User Interface, Adaptive Icons와 관련있는 항목으로 적한한 상황에서는 아이콘
    • 원형으로 출력되고 이외에서는 사각형 아이콘을 출력하게하는 속성
  • android:supportRtl
    • Right to Left 레이아웃 지원여부를 나태낸다.
  • android:theme
    • 앱의 테마를 설정 → android:theme="@style/App Theme"
    • 테마는 앱에서 사용되는 모양, 색깔 등을 지정하므로 변경되면 전체적인 느낌이 변경

##<activity>

  • Activity
    • Subclass는 App의 시각적 UI를 요소를 구현하는 역할을 한다.
    • 이러한 모든 Activity class는 mainfest file에서 <activity>로 나타나야 한다.
    • Activity class를 mainfest file에 <activity>로 나타내지 않으면 시스템은 Activity class를 실행하지 않는다
  • android:allowEmbedded
    • 액티비티가 다른 액티비티의 포함된 하위 항목으로 시작할 수 있음을 나타낸다.
  • android:allowTaskReparenting
    • 액티비티를 시작한 작업에서 해당 작업이 다음 앞으로 나올 때 액티비티를 선호하는 그다음 작업으로 이동할 수있는지를 나타낸다. 이값이 true인 경우 이동 가능 false인 경우 시작한 작업에 그대로 머물러 있어야한다.

##<provider>

  • <activity>와 동일하게 유일하게 사용하여야한다
    • 다른 App이 데이터베이스 접근이나 데이터를 공유하는것을 관리한다.
    • 기본적으로 다른 App의 data에 접근할 수 없기 때문에 Content Provider를 통해 접근을할 수 있게 허락한다.
    • 다른 App에 데이터를 제공해주는 Component이다. 그렇다고 아무데나 제공하는게 아니라 요청이들어와야한다.
  • Content
    • Provider가 data를 제공해주는데 요청은 Content Resolver class의 method로 할 수 있다.
    • 전화번호부와 카카오톡을 생각해보면 전화번호부에 등록외어 있는 전화번호 data가 자동으로 연동되어 카카오톡에 친구가 추가된다. 전화번호부에 Content Provider가 전화번호에 접근 할 수 있게 해주는것이고 카카오톡의 Content Resolver가 전화번호를 요청하여 친구 추가에 이용하는 것이다.
  • [android:authorities = "list"]가 반드시 포함되어야 한다. 안드로이드 시스템은 Content Provider의 Content Uri의 일부인 authority string에 따라 Content Provider를 참조하여 저장한다. → 전화번호에 돤련된 Content Provider에 접근한다고 하면 ContentReslover.query()를 사용해야한다. ContentReslover.query()는 provider를 식별하는 URI를 가져온다.

##<receiver>

  • Broadcasr Receiver를 사용하면 App의 다른 Component가 실행되고 있지 않아도
  • Android, os, ...App이 보내는 intents를 받을 수 있다.
  • OS, App이 작동하면서 어떤 이벤트가 발생하면 알림을 보내는데 이를 Broadcast라 한다.
  • 이벤트 발생 알림을 보낸다고 해서 OS, APP이 무조건 반응하지 않는다.
  • App도 필요한것만 시청, 받는다.
    • 요즘 문자로 인증번호가 오면 인증번호가 자동으로 입력되는 App이 있는데 이는 OS문자가 왔다고 BroadCast하면 해당 App이 이를 BroadCast Receiver를 통해 수신해서 일련의 과정을 거쳐 자동으로 입력되는것이다.
  • BroadCast Receiver
    • Static Receiver: mainfest에 <receiver>를 등록하느것이고
    • Dyamic Receiver: 특정 Componet에 context.registerReceiver()를 이용하여 등록
  • Component의 생명주기가 끝나면 사라진다.
  • Receiver 개수가 많아지면 배터리, 메모리 누수가 생기므로 Dynamic Reciver를 사용할 때
  • Receiver가 필요 없어지면 unregisterReciver()를 통해 해제해준다.
  • Receiver의 종류가 2가지인 이뉴느 앞서말한 메모리의 누수 때문이다.
  • Static Receiver는 한번 등록하면 계속 유지되는데 이게 단점이 될수 있다.
  • Receiver개수가 많아지면 차지하는 메모리가 많아지고 배터리도 빨리 방전되기 때문이다.
  • 계속 유지되어야하는 Receiver가 아니라면 Dynamic Receiver를 통해 등록과 해제를 적극적으로 활용하자
  • !Dynamic Receiver의 장점은 메모리 누수를 잡는것 뿐만 아니라 Component 내에 코드를 작성하기 때문에 해당 Component에 멤버변수를 활용하기 편하다.

##<Service>

  • <Activity>와 달리 시각적 UI를 가지지 않는다.
  • Service subclass는 다른 App에서 호출할 수 도있는 다양한 통신 API를 구현
  • long-running 백그라운드 작업에 사용된다.
  • service도 반드시 mainfestfile에 <service>로 나타내야 실행된다.
  • 백그라운드에서 구동되는 Component인 service는 한번 실행되면 App이 종료되어도 백그라운드에서 계속 실행된다.

##<intent-filter>

  • Component(Content Provider X)가 상황에 따라 다른 Component 실행이 필요하다면
  • 실행 할 Component에 대한 정보를 누군가(ActivityManager)에게 실행시켜달라고 요청해야한다.
  • Component에 대한 정보(이름)가 intent이다.
  • intent는실행될 Component에 전달해야 하는 data도 담을수 있다. (Intent는 목적의 뜻을 가지는데 IntentComponent를 실행하기 위한 의도를 전달하는 거지 직점 Component를 실행시키는게 아니기 때문이다라고 생가함)
  • Intent를 활용하여 Component를 활성한다.
  • Explict Intent(명시적 인텐트)
    • Component가 포함되어 있는 패키지명과 Component 이름을 정확하게 전달하는 경우
  • Implicit Intent(암시적 인텐트)
    • 패키지명과 이름을 전달하는게 아니라 Intent를 통해 Component가 수행 할 동작을 전달하는 경우
  • 동작 Component를 모두 보여준다. 2개이상일경우 사용자가 Component를 선택할 수 있게 한다.
  • <Intent-filter>는 암시적 인텐트를 사용하기 위해 작성한다.
  • 암시적 인텐트는 Component를 콕 찝어서 언급하는게 아니라 Component가 수행 할 동작을 전달한다.
  • 암시적 인텐트는 Component가 어떤 작업을 수행할 수 있는지, 어떤 데이터 유형을 처리할 수 있는지, 처리할 데이터의 주소는 무엇인지 등을 담고 있다.
  • Component중에서 Intent가 담고 있는 정보에 맞는 Component를 찾는 과정이 필요하다.
  • 인텐트해석(Intent Resolving)이라고 한다.
  • Component가 자신에게 해당하는 Intent의 조건이 무엇인지 mainfest file에 정의해두는데 이게 <intent-filter>이다.

(인텐트를 걸러주는 조건(필터)라고 생각..)

  • 컴포넌트 입장에사 생각해보면 수많은 인텐트 중에서 자신에게 맞는 인텐트만 받기 위해서 인텐트 필터 조건을작성해두는 것이다.

#암시적 인텐트가 존재하는 이유

- 안드로이드 OS에는 App간 경계가 없기 때문이다.

- 자기가 만든 App에서는 Package와 Component명을 알고 있으니 명시적 인텐트를 사용하면 되지만

내가 만든것이 아닌 App의 Component 명을 아는건 매우 어려운 일이다.

안드로이드 os는 Native App도 본이이 원하느 걸로 변경할 수 있는데

여기서 Native App의 Component를 명시적인텐트로 사용한다면

Native App을 변경한 사용자의 폰에서는 내가 만든 App이 똑바로 작동하지 않거나

다른 App으로 Native App을 대체하는게 힘들어 지기 때문이다.
